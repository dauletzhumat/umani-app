## 0. Легенда

### Размер (T-shirt sizing)

| Размер | Оценка | Смысл |
|---|---|---|
| **XS** | ≤ 0.5 дня | Тривиальная задача, обычно часть более крупной истории |
| **S** | 1 день | Одна задача по правилу CLAUDE.md ("максимум один рабочий день") |
| **M** | 2–3 дня | Декомпозируется на 2–3 задачи по 1 дню при спринт-планировании |
| **L** | 5–8 дней (1–1.5 недели) | Декомпозируется на 5+ задач по 1 дню |
| **XL** | 10+ дней (2+ недели) | **Сигнал**, что историю нужно раздробить на несколько отдельных Stories до включения в спринт — XL никогда не берётся в спринт как есть |

### Definition of Done (единый для всех историй, не повторяется в каждой карточке)

1. Реализовано согласно согласованному плану (план → подтверждение → код, правило CLAUDE.md)
2. Написаны unit-тесты на ключевую бизнес-логику (Jest — backend, `flutter test` — frontend)
3. Пройден код-ревью минимум одним членом команды
4. Обновлена соответствующая документация в `docs/*.md`, если история меняет контракт API/схему БД/UX-флоу
5. Учтены состояния Empty/Loading/Offline/Error там, где применимо ([05_UX.md §12–15](05_UX.md#12-empty-states))
6. Проверена локализация (RU/KZ/EN — строки не хардкожены)
7. Проходит CI (lint + tests + build) без ручных пропусков проверок
8. Задеплоено и вручную проверено в Staging
9. Продакт/QA принял историю по Acceptance Criteria

## 2. MVP — эпики (P0)

### Epic E1: Auth & Onboarding

**Story 1.1 — Splash Screen и выбор языка**
- Priority: P0 · Size: **XS**
- AC:
  - При холодном старте показывается Splash не дольше 1200мс воспринимаемо
  - Новый пользователь видит экран выбора RU/Қазақша/English
  - Выбор языка сохраняется и применяется ко всем экранам сразу

**Story 1.2 — Регистрация и вход по OTP**
- Priority: P0 · Size: **M**
- AC:
  - `POST /auth/register`/`login` отправляет OTP на телефон ([08_API.md §2](08_API.md#2-аутентификация-и-jwt))
  - `POST /auth/otp/verify` с верным кодом возвращает `accessToken`/`refreshToken`
  - Неверный код — ошибка `OTP_INVALID`, после 5 попыток — `TOO_MANY_ATTEMPTS`
  - UI показывает таймер повторной отправки и alternative "код не пришёл?"

**Story 1.3 — JWT + Refresh Token с ротацией**
- Priority: P0 · Size: **M**
- AC:
  - Access-токен истекает через 15 минут, клиент прозрачно обновляет через `/auth/refresh`
  - Каждый refresh выдаёт новую пару и отзывает предыдущий refresh-токен
  - Повторное использование отозванного refresh-токена отзывает все сессии пользователя

**Story 1.4 — Гостевой режим**
- Priority: P0 · Size: **S**
- AC:
  - На экране выбора авторизации доступна опция "Продолжить как гость"
  - Гостевая сессия работает локально (транзакции/бюджеты сохраняются), без синхронизации между устройствами
  - Есть путь апгрейда гостя в полный аккаунт без потери введённых данных

**Story 1.5 — Onboarding value-экраны**
- Priority: P0 · Size: **S**
- AC:
  - 3 свайпаемых экрана согласно [04_User_Flows.md §2](04_User_Flows.md#2-первый-запуск-и-onboarding)
  - Доступна кнопка "Пропустить" на каждом экране

**Story 1.6 — Первичная настройка аккаунта**
- Priority: P0 · Size: **S**
- AC:
  - Выбор базовой валюты (по умолчанию KZT)
  - Мультивыбор целей использования ("Понять траты", "Накопить", "Рассрочки" и т.д.) сохраняется в профиль

**Story 3.3 — OCR: интеграция Google Vision (backend)**
- Priority: P0 · Size: **L**

**Story 3.5 — AI автокатегоризация**
- Priority: P0 · Size: **M**
- AC: транзакция без явно указанной категории получает предложенную AI-категорию; при `confidence < 0.5` категория не проставляется автоматически ([10_AI.md §9](10_AI.md#9-prompt-engineering)); повторный мерчант+сумма — из кэша Redis, не повторный вызов AI

**Story 3.6 — Лента транзакций с фильтрами**
- Priority: P0 · Size: **M**
- AC: фильтры по счёту/категории/типу/периоду; курсорная пагинация ([08_API.md §4](08_API.md#4-pagination)); свайп строки — быстрая смена категории

## 8. Итог
